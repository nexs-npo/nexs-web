# nexs ナレッジシステム 構造ガイド

このドキュメントは、nexs Knowledge ページの記事作成における基本的な考え方と構造を説明します。

## 概要

nexs ナレッジシステムは、社会課題の解決に向けた**研究と実装の循環プロセス**を可視化し、共有するための知識管理システムです。

5つのカテゴリに分けることで、抽象的な「知」から具体的な「行動」「データ」「学習」へと流れていく構造になっています。

---

## 5つのカテゴリ

### 1. **基礎知識（FOUNDATION）** #F-XXX

**定義：**
社会課題解決の基盤となる理論・概念・背景知識

**役割：**
- 仮説を立てる際の前提となる知識を提供する
- 複数の仮説の源泉となり得る
- 普遍的で、時間が経っても変わりにくい内容

**含むもの：**
- 理論・学説・思想
- 背景となる社会的文脈
- 既存研究のサマリー

**含まないもの：**
- 特定のプロジェクト固有の内容
- 実装方法の詳細

---

### 2. **仮説（THESIS）** #T-XXX

**定義：**
基礎知識から導き出した「社会課題のあり方」「解決の道筋」についての問い・提案

**役割：**
- 「何が課題か」「なぜそれが課題か」を定義する
- プロトコル設計の指針となる

**含むもの：**
- 課題設定・問い立て
- 仮説の背景と論理
- 基礎知識への参照

**含まないもの：**
- 具体的な実装手順
- 測定方法の詳細

---

### 3. **プロトコル（PROTOCOL）** #P-XXX

**定義：**
仮説を検証するための具体的な計画・実装方法・評価基準

**役割：**
- 仮説を「試してみるもの」に変える
- 再現可能な形で設計を記述する
- エビデンス収集の枠組みを示す

**含むもの：**
- 実装計画（期間・対象・リソース）
- 具体的な手法・ステップ
- 測定・評価の指標と方法

**含まないもの：**
- 実装中の試行錯誤
- 未確定な数値データ

---

### 4. **エビデンス（EVIDENCE）** #E-XXX

**定義：**
プロトコルに基づいて実装した結果のデータ・観察・発見

**役割：**
- 仮説の有効性を検証する
- 次の課題を浮彫りにする

**含むもの：**
- 収集されたデータ・統計
- 定性的な観察・事例
- 想定との差異・予期しない発見

**含まないもの：**
- 解釈や分析は分けて記述
- 将来の提言

---

### 5. **アップデート（UPDATE）** #U-XXX

**定義：**
エビデンスから導き出された新たな知見・提言・次のサイクルへの入り口

**役割：**
- 実装から学習へ
- 閉じた経験を社会への提言に変える
- 新たな仮説のための基礎知識へ

**含むもの：**
- 分析結果・考察
- 社会への提言・知見の応用可能性
- 次のサイクルへの提起

**含まないもの：**
- 既に基礎知識や他のカテゴリで述べた内容の繰り返し

---

## カテゴリ間の流れ

```
基礎知識 (F)
    ↓
   仮説 (T) ← 基礎知識を参照、「課題とは何か」を問う
    ↓
プロトコル (P) ← 仮説を参照、「どう検証するか」を示す
    ↓
エビデンス (E) ← プロトコルを参照、データを記述
    ↓
アップデート (U) ← エビデンスを参照、「何が学べたか」を述べる
    ↓
新しい基礎知識へ
```

**重要：** 各カテゴリは、前のカテゴリを参照することで、「何が前提だったか」を明確にする。

---

## 参照とリンク戦略

### 参照の方向性

- **T → F：** 「この仮説が基づいている基礎知識」を引用
- **P → T：** 「この仮説を検証するための計画」として機能
- **E → P：** 「このプロトコルで得たデータ」として記述
- **U → E：** 「このエビデンスから学べたこと」として解釈

### 参照の形式

記事本文内で「`→ #F-001`」のような形で、他のカテゴリへのリンクを示す。

**例：**
```
この考え方は → #F-001 の「適者生存の誤読」の議論を基にしています。
```

## 階層化・冗長性を避けるための基本原則

### 原則1：「一つのアイデアは一箇所に」
同じ内容は複数の記事に書かない。一箇所にまとめて、他の記事では参照リンクを張る。

### 原則2：「各カテゴリの役割を明確に」
カテゴリをまたいで同じレベルの内容を書かない。
- 「仮説」の記事に「実装方法」を書かない
- 「プロトコル」の記事に「理論背景」を詳しく書かない

### 原則3：「参照を活かす」
「これについてはこちらを読んでください」という参照を、読者への負担ではなく、全体構造への案内として機能させる。

### 原則4：「時間軸を意識する」
各記事は、執筆時点での「決定事項」「確定したデータ」のみを含める。
検討中・試験段階の内容は時期尚早。

### 原則5：「背景と本体を分ける」
記事の本体は簡潔に。背景説明が長くなる場合は、「参照：#F-XXX」として別の記事に任せる。

### 原則6：「誰のための記事か」を明確に」
- 内部向け（次の仮説立案のため）か
- 外部向け（社会への提言として）か
で、トーンや詳細度を変える。

### 原則7：「更新性を考える」
一度書いた内容が古くなったら、新しいカテゴリの記事を追加する。古い記事は削除せず、「参照：新しい#U-XXX」と更新を示す。

## カテゴリ別 テンプレート

### 基礎知識（FOUNDATION）

```markdown
# #F-XXX：[タイトル]

## 概要
[この基礎知識が何を説明しているかを簡潔に]

## 背景
[なぜこの知識が重要か、どういう文脈で考えられるようになったか]

## 主要な考え方
[複数のセクションに分けて、段階的に説明]

## 応用可能性
[この知識がどのような仮説や課題に応用できるか、簡潔に]

## 参考資料
- [外部リソース、論文など]
```

### 仮説（THESIS）

```markdown
# #T-XXX：[仮説のタイトル]

## 課題設定
[「何が課題か」を明確に]

## 仮説
[解決の方向性・アプローチの提案]

## 基礎となる知識
- 参照：#F-XXX
- 参照：#F-YYY

## この仮説から導ける検証方法
[どうやってこれを試すのか、大まかな方向性]
```

### プロトコル（PROTOCOL）

```markdown
# #P-XXX：[プロトコルのタイトル]

## 目的
[この実装が検証したい仮説は #T-XXX]

## 実装計画
[期間、対象、リソース、主なステップ]

## 具体的な方法
[手順を段階的に]

## 評価指標・測定方法
[何をもって「成功」と判断するか]

## リスク・制約
[想定される課題、環境制約など]
```

### エビデンス（EVIDENCE）

```markdown
# #E-XXX：[エビデンスのタイトル]

## プロトコル
[この実装は #P-XXX に基づいています]

## 収集されたデータ
[統計情報、測定結果、観察内容]

## 定性的な発見
[事例、インタビュー、フィールドノート]

## 想定との差異・予期しない発見
[データが示したサプライズ]
```

### アップデート（UPDATE）

```markdown
# #U-XXX：[アップデートのタイトル]

## エビデンスの要約
[#E-XXX のデータから何が分かったか]

## 分析・考察
[データが意味するところ]

## 社会への提言・応用可能性
[この学習が他の地域や組織でどう活かせるか]

## 次のサイクルへ
[新たに必要な基礎知識、新たな仮説候補など]
```

## 記事作成の流れ

1. **どのカテゴリの記事を書くか、決める**
   - 既存の記事との関係を確認
   - 必要な参照元は何か

2. **テンプレートに沿って執筆**
   - 他のカテゴリの内容を混ぜない
   - 参照はリンク形式で明示

3. **冗長性をチェック**
   - 同じ内容が他の記事にないか確認
   - あれば参照に変更

4. **カテゴリ全体での位置付けを確認**
   - この記事は「循環」のどこに位置するか
   - 次のカテゴリへの流れは明確か

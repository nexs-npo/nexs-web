# 16. AI-Native Coding Guide

**Version:** 1.0
**Created:** 2026-02

本ドキュメントは **[01_PHILOSOPHY.md](./01_PHILOSOPHY.md)** の Level 2（Method）および Level 3（Assurance）を受けて、コードレベルの実践を定義する。

## Part 1: コード記述の実践（Level 2 を受けて）

### 1-1. 記述スタイル

**過小に書く。必要になったら足す。**

- 関数・コンポーネントは1つの責務だけを持つ。「〜して、さらに〜する」という処理は分割のサイン。
- コメントは「何をしているか」ではなく「なぜそうしているか」を書く。コードで読めることを文章にしない。
- 型は`any`を使わない。不明な場合は`unknown`とし、型ガードで絞り込む。
- マジックナンバー・マジック文字列は定数に抽出する。ただし1箇所にしか現れないなら直書きでよい。
- エラーハンドリングはシステム境界（ユーザー入力、外部API呼び出し）でのみ行う。内部関数は失敗を呼び出し元に伝播させる。

**やらないこと（over-engineering の兆候）：**

- 将来のユースケースのための汎用化
- 2〜3箇所の類似コードの共通化（重複は許容する）
- 使われていない引数・オプションの追加
- 変更していないコードへのdocstring/コメント追加

### 1-2. ファイル・モジュール設計

**Locality of Context の具体的な実現。**

- 1ファイルは1つの関心事。読者がそのファイルだけを見て意図を把握できる状態を目指す。
- `import`の深さは最大3階層（`../../..`以上は設計の見直しサイン）。
- TypeScriptのパスエイリアス（`@/*`）を使い、相対パスの深さを隠蔽する。
- ページ固有のコンポーネントはそのページのディレクトリに置く。複数ページで使うものだけ`src/components/`に昇格させる。

### 1-3. 依存管理

**依存を増やすコストは、解決されるコストより大きいことが多い。**

- 新しいnpmパッケージを追加する前に、標準APIやすでに導入済みのライブラリで解決できないかを確認する。
- ライセンスを確認する。AGPL等の「感染性」ライセンスは、OSS公開前提のこのプロジェクトでも商用利用の可否を精査する。
- パッケージのメンテナンス状況（最終更新日、GitHub Stars、Issues）を確認する。
- 依存を追加する際は`package.json`の`dependencies`/`devDependencies`を正しく分類する。

### 1-4. AIとの協働プロセス

**AIは優秀な実装者だが、文脈を誤解することがある。レビューの軸を「コードの行」から「振る舞い」に移す。**

作業の流れ：

```
1. タスクの範囲を明確にする（どこを変えて、どこを変えないか）
2. AIに実装させる
3. 変更されたファイルを差分（git diff）で確認する
4. 動作を検証する（次のPart 2参照）
5. 問題がなければコミット
```

AIへの指示で効果的なパターン：

- 「このファイルだけを変更してください」と範囲を明示する
- 「既存のコードスタイルに合わせてください」と制約を伝える
- 「変更前後の差分を教えてください」と確認を求める
- 長い作業は「ここまでコミットしますか？」で区切り、安全なセーブポイントを作る

AIが生成したコードを信頼しないパターン：

- テストを書かずに「動いているはず」と思い込む
- 差分を読まずにそのまま適用する
- エラーが出たときに「AIが直してくれるはず」と同じプロンプトを繰り返す（アプローチを変える）

## Part 2: 検証の実践（Level 3 を受けて）

### 2-1. 検証ツール（現行スタック）

| ツール | 役割 | 実行タイミング |
|--------|------|--------------|
| TypeScript | 型検査 | 開発中（IDE）/ CI |
| Biome | Lint + Format | 開発中（IDE）/ CI |
| `npm run build` | ビルド検証 | コミット前 / CI |
| — | 自動テスト（未導入） | — |

**現状の制約：** 自動テストフレームワーク未導入。影響の大きい変更（認証、データベース操作、APIレスポンス）については手動での動作確認を必須とする。テストフレームワーク導入時はこの表を更新する。

### 2-2. コミット前チェックリスト

最低限、コミット前に以下を確認する：

```bash
npm run build   # ビルドエラーがないことを確認
```

ビルドが通らない状態でコミットしない。`wip:` プレフィックスで途中保存する場合も例外ではない。

### 2-3. 影響の大きい変更の検証基準

以下の種類の変更は、自動検証に加えて手動での動作確認を行う：

- **認証フロー変更：** ログイン・ログアウト・ロール制御の確認
- **データベース操作：** RLS（行レベルセキュリティ）の意図通りの動作確認
- **APIエンドポイント変更：** レスポンス形式・エラーハンドリングの確認
- **外部サービス連携：** Webhook、OAuth、サードパーティAPIとのやりとり

### 2-4. 動作確認の記録

バグ修正や複雑な機能追加では、確認した内容を`.work/DEV_LOG.md`に記録する。

記録項目の目安：

- 何を確認したか（テスト観点）
- 確認方法（手動操作の手順、実行したコマンド）
- 結果（期待通りか、異常がなかったか）
- 発見したバグと修正内容

記録の目的は「後から同じことを再現できる状態を保つ」こと。

### 2-5. セキュリティチェック観点

AIが生成したコードで見落としやすい脆弱性を重点的に確認する：

- **環境変数の露出：** `PUBLIC_` プレフィックスのない秘密情報がクライアントサイドに渡っていないか
- **認証バイパス：** APIルートで認証チェックをスキップしていないか
- **入力検証の欠如：** ユーザー入力をサニタイズせずにDBクエリやコマンドに使っていないか
- **依存関係の脆弱性：** `npm audit` で既知の脆弱性がないか（定期的に実行）

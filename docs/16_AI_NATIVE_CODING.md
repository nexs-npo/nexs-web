# 16. AI-Native Coding Guide

**Version:** 0.1 (Draft)
**Status:** 🚧 草稿 — 手探りしながら作り込む。破壊的変更あり。
**Created:** 2026-02

本ドキュメントは **[01_PHILOSOPHY.md](./01_PHILOSOPHY.md)** の Level 2（Method）および Level 3（Assurance）を受けて、コードレベルの実践を定義する。

原則の「なぜ」は哲学書を参照すること。本書は「どうやって」に集中する。

---

## 用語定義

**Snapshot（スナップショット）：** AIに渡す最小の「理解単位」。原則としてファイル単位。

**Blast Radius（爆発半径）：** 変更が波及する範囲。小さいほど良い。

**Anchor（アンカー）：** AIの生成・変更のブレを抑える制約（型、テスト、アサーション、Lint）。

---

## 1. Snapshot First — 文脈はスナップショットで完結させる

### 1.1 関連する文脈を「一つの場所」に寄せる

1つの関心事（use case / feature / domain rule）は、できるだけ少ないファイルで完結させる。
読み手（AI/人）が複数ファイルを横断して「暗黙の接続」を推測する設計を避ける。

- 1ファイルは1つの関心事。読者がそのファイルだけを見て意図を把握できる状態を目指す。
- `import` の深さは最大3階層（`../../..` 以上は設計の見直しサイン）。
- TypeScript のパスエイリアス（`@/*`）を使い、相対パスの深さを隠蔽する。
- ページ固有のコンポーネントはそのページのディレクトリに置く。複数ページで使うものだけ `src/components/` に昇格させる。

### 1.2 過度な分割・過度な抽象化を避ける

小さく分けすぎると、AIは「いま必要な文脈」を欠いて誤修正しやすい。
分割する場合は、境界が明確で、契約（interface）が明示されていること。

---

## 2. Explicit over Implicit — 推測させない

### 2.1 暗黙のドメイン知識を排除する

「読めば分かるはず」「常識で分かるはず」を前提にしない。
意図は型・命名・コメント（必要最小限）で固定する。

- コメントは「何をしているか」ではなく「なぜそうしているか」を書く。コードで読めることを文章にしない。
- 型は `any` を使わない。不明な場合は `unknown` とし、型ガードで絞り込む。
- エラーハンドリングはシステム境界（ユーザー入力、外部API呼び出し）でのみ行う。内部関数は失敗を呼び出し元に伝播させる。

### 2.2 命名はセマンティクスを優先する

省略・隠語・ローカルジョークを避ける。
変数・関数・型名は「何を表すか」が単体で分かること。

### 2.3 インターフェース（契約）を明示する

入力・出力・エラー・副作用の形を明示する（型や戻り値の形で表現する）。
"nullっぽい何か" "例外が出るかもしれない" を曖昧にしない。

---

## 3. Predictability over Elegance — 短さより予測可能性

### 3.1 退屈で定石通りのコードを選ぶ

気の利いたワンライナーより、読み下せる定石を優先する。
トリック（暗黙変換、凝った演算子、条件の詰め込み）で行数を減らさない。

**やらないこと（over-engineering の兆候）：**

- 将来のユースケースのための汎用化
- 使われていない引数・オプションの追加
- 変更していないコードへの docstring/コメント追加

### 3.2 スタイルは一貫させる

同じ問題には同じパターンで答える（分岐、エラー処理、命名、層の分け方）。
ばらつきはAIの誤推測を増やす。

---

## 4. Static over Dynamic — 動的な魔法を避ける

### 4.1 実行時に形が変わる設計は避ける

メタプログラミング、リフレクション、動的ディスパッチ等は、AIが静的に追えず誤修正の温床になる。
静的に追える形を第一候補にする。

### 4.2 どうしても必要なら境界に隔離する

動的要素が必要な場合は、コアロジックから隔離し、境界を薄く保つ。
境界にはアンカー（型・テスト）を厚く置く。

---

## 5. WET is Allowed — DRYより局所性（ただし条件付き）

### 5.1 重複は「被害局所化」のために許容される

過度な共通化は依存関係の連鎖を生み、爆発半径を広げる。
AIが「そのファイルだけ」で安全に変更できることを優先し、2〜3箇所の類似コードの重複を許容する。

### 5.2 共通化は"安定してから"検討する

共通化は慎重に行う。導入するときは「変更頻度」「影響範囲」「テストで守れるか」を根拠にする。
共通化した場合は、利用側の文脈が失われないよう契約を厚くする（型・テスト・例）。

目安として、3回以上の重複が確認され、かつ変更頻度が低い場合にのみ検討する。

---

## 6. Anchors First — 制約を先に置く

### 6.1 型・テスト・Lintは「生成のアンカー」

それらは事後検証だけでなく、AIの生成・変更のブレを防ぐガードレールとして機能する。
変更・追加の前に、期待する振る舞いをアンカーとして定義することを優先する。

### 6.2 境界ほどアンカーを厚くする

認証、課金、権限、入力、永続化など「壊れると致命傷」の箇所は特に厚く守る。
正常系だけでなく異常系・境界条件・悪用シナリオも含めて固定する。
権限境界や外部公開面では、必要に応じて攻撃テスト（レッドチーム的テスト）を行い、「壊れない」ことを確認する。

---

## 7. Dependency Hygiene — 依存は少なく、標準で、追えるもの

### 7.1 未知・独自・薄いドキュメントの依存は避ける

AIも人も追えない依存はハルシネーションの温床になる。
「標準」「世界標準」「枯れている」「ドキュメントが厚い」を優先する。

- 新しい npm パッケージを追加する前に、標準 API や導入済みのライブラリで解決できないかを確認する。
- パッケージのメンテナンス状況（最終更新日、GitHub Stars、Issues）を確認する。
- 依存を追加する際は `package.json` の `dependencies`/`devDependencies` を正しく分類する。

### 7.2 依存導入は"主権"を守る

ベンダーやライブラリに依存する場合は、移行可能性・データ可搬性・代替手段を意識する。
依存を増やす時は、爆発半径とアンカーをセットで考える。

- ライセンスを確認する。AGPL 等の「感染性」ライセンスは、OSS 公開前提のこのプロジェクトでも商用利用の可否を精査する。

## 8. Change Protocol — AIと協働する変更の作法

### 8.1 AIに渡すのはDiffではなくSnapshot

差分だけだと文脈が欠けやすい。原則としてファイル全体（または関心事が完結する単位）を渡す。
変更の意図・制約・期待する結果（アンカー）を同時に渡す。

AIへの指示で効果的なパターン：

- 「このファイルだけを変更してください」と範囲を明示する
- 「既存のコードスタイルに合わせてください」と制約を伝える
- 長い作業は「ここまでコミットしますか？」で区切り、安全なセーブポイントを作る

### 8.2 変更は"検証可能な単位"に分割する

大きすぎる変更は、人にもAIにも検証が難しい。
1変更 = 1意図を守り、各単位がテストやチェックで閉じるようにする。

作業の流れ：

```
1. タスクの範囲を明確にする（どこを変えて、どこを変えないか）
2. AIに実装させる（ファイル全体を渡す）
3. 変更されたファイルを差分（git diff）で確認する
4. アンカー（型・Lint・ビルド）を通過させる
5. 動作を検証する（Appendix 参照）
6. 問題がなければコミット
```

### 8.3 生成コードはアンカー通過が統合条件

生成したものを「読んだ気がする」で統合しない。
型・テスト・Lint・スキャン等の通過を根拠に統合する。

AIが生成したコードを信頼しないパターン：

- テストを書かずに「動いているはず」と思い込む
- 差分を読まずにそのまま適用する
- エラーが出たときに同じプロンプトを繰り返す（アプローチを変える）

## Appendix: 現行スタックでの実践

本セクションは現在のプロジェクト構成（Astro / TypeScript / Biome）に固有の実践を記す。
技術選定が変わった際はこの Appendix を更新し、上記ルール（1-8）には手を加えない。

### A-1. 現行アンカーツール

| ツール | 役割 | 実行タイミング |
|--------|------|--------------|
| TypeScript | 型検査 | 開発中（IDE）/ CI |
| Biome | Lint + Format | 開発中（IDE）/ CI |
| `npm run build` | ビルド検証 | コミット前 / CI |
| — | 自動テスト（未導入） | — |

**現状の制約：** 自動テストフレームワーク未導入。影響の大きい変更（認証、データベース操作、APIレスポンス）については手動での動作確認を必須とする。テストフレームワーク導入時はこの表を更新する。

### A-2. コミット前チェックリスト

最低限、コミット前に以下を確認する：

```bash
npm run build   # ビルドエラーがないことを確認
```

ビルドが通らない状態でコミットしない。`wip:` プレフィックスで途中保存する場合も例外ではない。

### A-3. 影響の大きい変更の検証基準

以下の種類の変更は、自動検証に加えて手動での動作確認を行う（Rule 6.2 の境界ほど厚く守る原則に対応）：

- **認証フロー変更：** ログイン・ログアウト・ロール制御の確認
- **データベース操作：** RLS（行レベルセキュリティ）の意図通りの動作確認
- **APIエンドポイント変更：** レスポンス形式・エラーハンドリングの確認
- **外部サービス連携：** Webhook、OAuth、サードパーティAPIとのやりとり

### A-4. 動作確認の記録

バグ修正や複雑な機能追加では、確認した内容を `.work/DEV_LOG.md` に記録する。

記録項目の目安：

- 何を確認したか（テスト観点）
- 確認方法（手動操作の手順、実行したコマンド）
- 結果（期待通りか、異常がなかったか）
- 発見したバグと修正内容

記録の目的は「後から同じことを再現できる状態を保つ」こと。

### A-5. セキュリティチェック観点

AIが生成したコードで見落としやすい脆弱性を重点的に確認する：

- **環境変数の露出：** `PUBLIC_` プレフィックスのない秘密情報がクライアントサイドに渡っていないか
- **認証バイパス：** APIルートで認証チェックをスキップしていないか
- **入力検証の欠如：** ユーザー入力をサニタイズせずにDBクエリやコマンドに使っていないか
- **依存関係の脆弱性：** `npm audit` で既知の脆弱性がないか（定期的に実行）

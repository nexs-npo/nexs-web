# briefs/

**作業ログではありません。**
指示・実装の意図・レビューの根拠を次の作業者に渡すための引き継ぎ文書です。

コードの中身ではなく「**なぜそう判断したか**」を記録し、三者（ユーザー・ClaudeCode・Codex）が同じ文脈を共有できる状態を維持します。コードを読まなくても意思決定の経緯を追跡できることが目的です。

---

## ワークフロー

```
1. User → ClaudeCode  : 指示（## Task を記入）
2. ClaudeCode         : 作業 → コミット + ## Implementation を記入
3. Codex              : ## Implementation を読む → ## Review を記入
4. User + ClaudeCode  : ## Review を読んで議論 → ## Resolution を記入
5. 完了               : ファイル名の日付を確定、Status を completed に変更
6. Push → squash merge
```

---

## ファイル命名規則

```
draft-{ブランチ名}.md        # 作業中
YYYY-MM-DD-{ブランチ名}.md   # 完了時にリネーム
```

例：`draft-feat-signing-flow.md` → `2026-02-21-feat-signing-flow.md`

---

## テンプレート

```markdown
# {ブランチ名}

**Branch:** {branch-name}
**Status:** draft | completed
**Date:** YYYY-MM-DD

## Task
<!-- ユーザーの指示を転記または要約する -->
<!-- 「なぜこのタスクが必要か」の背景も書くと良い -->

## Implementation
<!-- ClaudeCode が記入 -->
<!-- 何をしたか（What）より、なぜその設計を選んだか（Why）を優先する -->
<!-- 参照コミット: abc1234 -->

## Review
<!-- Codex が記入 -->
<!-- 意図が設計原則と整合しているかを中心にレビューする -->
<!-- コードの細部より「判断の根拠」に焦点を当てる -->

## Resolution
<!-- User + ClaudeCode が記入 -->
<!-- レビューへの対応内容、または却下した場合はその理由 -->
```

## 記述ルール

### 書くこと
- **Why（なぜ）**: この設計を選んだ理由、却下した代替案とその理由
- **Trade-off**: 何を得て何を手放したか
- **前提条件**: この判断が成立する条件（外れたら再検討が必要なもの）

### 書かないこと
- **What（何をしたか）の詳細**: それはコミットメッセージと git diff が担う
- **How（どう実装したか）の詳細**: コードを読めば分かることを文章にしない
- **未対応の脆弱性・セキュリティ上の抜け**: 公開リポジトリに書くと攻撃の手がかりになる。セキュリティ上の懸念はプライベートな手段（口頭・非公開Issue等）で共有する

### セキュリティ上の書き方の基準

> 「この判断を採用した理由」は書いてよい。「採用しなかった対策と、その対策が無い箇所」は書かない。

## このディレクトリについて

- ファイルはすべて公開資産（GitHub上で誰でも閲覧可能）
- 過去のファイルは変更しない（Resolution 記入後は追記のみ）
- ブランチ1本 = ファイル1つ を原則とする

# briefs/

**作業ログではありません。**
指示・実装の意図・レビューの根拠を次の作業者に渡すための引き継ぎ文書です。

コードの中身ではなく「**なぜそう判断したか**」を記録し、関係者が同じ文脈を共有できる状態を維持します。コードを読まなくても意思決定の経緯を追跡できることが目的です。

各エントリの記入者は、末尾に**署名**を残します。署名により「誰が・いつ・どの立場で書いたか」が明確になります。

---

## ワークフロー

```
1. 指示者       : 指示（## [date] Task を記入・署名）
2. 作業担当     : 作業 → コミット + ## [date] Implementation を記入・署名
3. レビュー担当 : スクリプトを実行 → ## [date] Review — {type} が追記される
                  ※ 複数のレビュー担当が独立して追記してよい
4. 作業担当     : レビューを受けて修正 → ## [date] Implementation — 修正 を追記・署名
5. （3-4を繰り返す）
6. 指示者 + 作業担当 : ## Resolution を記入・署名
7. 完了         : ファイル名の日付を確定、Current state を completed に変更
8. Push → squash merge
```

担当者（人・AI）は状況に応じて変わりうる。特定のツールやモデルに固定しない。

---

## レビューの種類とタイミング

| スクリプト | 焦点 | 使うタイミング |
|---|---|---|
| `scripts/review-codex-philosophy.sh` | 意図が `01_PHILOSOPHY.md` の Level 0–3 と整合しているか | 設計判断をした直後 |
| `scripts/review-codex-logic.sh` | コードが実装意図通りに動いているか・ロジックの穴 | 実装中、動作しない箇所がある時 |
| `scripts/review-codex-security.sh` | 公開前に危険なものが含まれていないか | push 直前 |

## 署名フォーマット

```
*Signed: {名前またはツール名} ({識別子}) — {YYYY-MM-DD}*
```

例：
```
*Signed: Claude Code (claude-sonnet-4-6) — 2026-02-21*
*Signed: Codex (codex-1) — 2026-02-21*
*Signed: Gemini CLI (gemini-2.5-pro) — 2026-02-21*
*Signed: 山田太郎 (human) — 2026-02-21*
```

## ファイル命名規則

```
draft-{ブランチ名の / を - に置換}.md   # 作業中
YYYY-MM-DD-{同上}.md                    # 完了時にリネーム
```

例：`draft-feat-signing-flow.md` → `2026-02-21-feat-signing-flow.md`

## テンプレート

```markdown
# {ブランチ名}

**Branch:** {branch-name}
**Status:** draft | completed
**Current state:** （最新の状態を1行で。AIが全文読まなくても現在地を把握できるようにする）
**Date:** YYYY-MM-DD

## [YYYY-MM-DD] Task
<!-- 指示を転記または要約する。「なぜこのタスクが必要か」の背景も書くと良い -->

*Signed: {名前} ({識別子}) — {date}*

## [YYYY-MM-DD] Implementation
<!-- 何をしたか（What）より、なぜその設計を選んだか（Why）を優先する -->
<!-- 参照コミット: abc1234 -->

*Signed: {名前} ({識別子}) — {date}*

## [YYYY-MM-DD] Review — philosophy | logic | security
<!-- レビュー担当が記入。スクリプトで自動追記される -->

*Signed: {名前} ({識別子}) — {date}*

## [YYYY-MM-DD] Implementation — 修正（任意）
<!-- レビューを受けた修正の意図。繰り返し追記してよい -->

*Signed: {名前} ({識別子}) — {date}*

## Resolution
<!-- 完了時に記入。レビューへの最終的な対応・却下理由など -->

*Signed: {名前} ({識別子}) — {date}*
```

## 記述ルール

### 書くこと
- **Why（なぜ）**: この設計を選んだ理由、却下した代替案とその理由
- **Trade-off**: 何を得て何を手放したか
- **前提条件**: この判断が成立する条件（外れたら再検討が必要なもの）

### 書かないこと
- **What（何をしたか）の詳細**: それはコミットメッセージと git diff が担う
- **How（どう実装したか）の詳細**: コードを読めば分かることを文章にしない
- **未対応の脆弱性・セキュリティ上の抜け**: 公開リポジトリに書くと攻撃の手がかりになる。セキュリティ上の懸念はプライベートな手段（口頭・非公開Issue等）で共有する

### セキュリティ上の書き方の基準

> 「この判断を採用した理由」は書いてよい。「採用しなかった対策と、その対策が無い箇所」は書かない。
